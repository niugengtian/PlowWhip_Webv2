## 1. 执行结论

项目已经形成一个结构清楚、可构建、具备较强本地可恢复性的 Docker-first MVP：SQLite 是任务真源，任务状态和修订受事务保护；全局 Cron、任务/资源租约、Host Job 双账本、上下文编译、确定性验证、审计、备份和控制台均有实际代码及测试。当前基线通过全部既有质量门，真实 HTTP 链路也能完成两个项目的严格档任务。

但不能把它判定为“Token 预算严格受控、权限已强制执行、严格质量门独立、最大并发硬受限”的完整无人值守系统。定向复现证明：

1. `token_budget=0` 的 Host Provider 任务仍会被实际派发；
2. `max_parallel_workers=1` 时，连续两个 Tick 可留下 2 个同时运行的 Host Job；
3. 权限授权/拒绝记录没有接入任何任务领取、命令执行、网络或秘密引用决策；
4. Convention 模型精炼不进入统一 Token 账本和全局预算；
5. “strict 独立审查”只是用同一确定性验证器立即重复相同检查，不是独立实现、独立进程或模型审查；
6. 完成证据可由任务执行前已经存在的文件满足，证据哈希也不包含被验证文件的内容哈希。

因此，本报告的总体判断是：

- **作为单机、可信操作者、低风险项目的本地 MVP：基本可用。**
- **作为可能产生模型费用、长期无人值守且依赖硬预算/硬权限/强完成证据的系统：尚不满足可靠投产条件。**
- **作为不可信代码沙箱、多租户平台或公网服务：明确不适用，项目文档也部分承认这一边界。**

## 3. 设计思维与系统架构

### 3.1 核心设计思想

从 `README.md:3-5`、`docs/WEB_V2_DESIGN.zh-CN.md:19-34` 和 `docs/ARCHITECTURE.md:3-18` 可归纳出五个主要原则：

1. **目标驱动而非聊天驱动**：任务、预算、状态和证据进入数据库，不把聊天历史当真源。
2. **确定性控制面**：调度、租约、网络探测、恢复和验证原则上不调用模型。
3. **上下文编译而非全历史回放**：Objective、Role 和三层 Convention 被编译为有大小上限的 Context Pack。
4. **稳定角色会话**：Project → Role → Worker 持久绑定，外部 CLI session 与内部 binding 分离。
5. **证据决定完成**：任务只有经过 Verification Engine 才能进入 completed。

这些原则彼此一致，且明显针对 Token 浪费、会话漂移、重复派发和模型自报完成等常见 Agent 风险。

### 3.2 实际分层

实际代码形成以下结构：

- **展示/控制层**：React 单页控制台与 FastAPI 路由，入口为 `web/src/App.tsx` 和 `backend/plow_whip_web/api/app.py`。
- **应用服务层**：`TaskService`、`SchedulerService`、`RecoveryService`、`BudgetManager`、`ContextCompiler`。
- **领域/持久化层**：`TaskStatus`、Repository 类、SQLite WAL 和顺序迁移。
- **执行层**：
  - 容器内 `GenericCommandProvider`；
  - 宿主机受限 Host Bridge；
  - Codex、Cursor 和 JSON simple-worker 固定适配器。
- **观察/运维层**：task events、audit log、outbox/SSE、health、backup、diagnostics。

优点是依赖方向总体清楚，Provider 不直接拥有任务状态；不足是 `api/app.py` 同时承担依赖装配、认证、审计和大量路由，当前 523 行，已开始成为组合根与业务编排混合的单体模块。

### 3.3 数据一致性模型

SQLite 连接启用 foreign keys、WAL 和 5 秒 busy timeout（`store/database.py:13-19`）。任务领取使用 `BEGIN IMMEDIATE`、expected revision、幂等事件键、任务租约、资源锁和项目 fencing token（`store/task_repository.py:182-248`、`548-609`）。这对单机部署是合理且务实的方案。

局限：

- 迁移只记录文件名，不保存 SQL checksum；已应用迁移被改写时不会检测漂移（`store/database.py:47-60`）。
- 多个状态字段主要靠应用逻辑约束，数据库缺少 task/worker/status 枚举 CHECK。
- 全局 scheduler fencing token 没有传到 Host Bridge 执行请求，也没有被 Bridge 校验；稳定 job id 才是实际执行侧的幂等边界。

## 5. 主要优点

### 5.1 状态权威和执行者分离

任务状态只由 Repository/Service 修改，模型/CLI 只返回执行结果。领取、验证、完成分别有 revision 和状态前置条件，避免“模型输出 done 就完成”。这是项目最扎实的设计。

### 5.2 Host Job 对重复派发采取保守策略

容器数据库和 Host Bridge 状态文件组成双账本。Bridge 在创建进程前写 `dispatching`，随后持久化 PID/process identity/session（`host_bridge.py:147-202`）；容器恢复会排除未消费 Host Job（`runtime/recovery.py:21-27`）。不能确认旧进程死亡时不盲目重派，方向正确。

### 5.3 本地运行安全基线较好

Compose 只发布 `127.0.0.1`，容器使用非 root 用户、只读根文件系统、drop all capabilities 和 `no-new-privileges`（`compose.yaml:10-29`）。Host Bridge 有 Bearer token、项目根 allowlist、固定 adapter 和环境 allowlist。Generic Command 不使用 shell 且环境变量受限。

### 5.4 故障不会轻易拖垮全局 Tick

任务并发执行时单任务异常被收敛为 deferred，SQLite locked/busy 被安全跳过，runner 捕获一次 Tick 异常后继续。对单机长期运行有实际价值。

### 5.5 可维护的测试基础

后端 168 项测试覆盖状态机、迁移、调度、Cron、恢复、并发、Host Job、安全边界、备份和 E2E。测试大量使用临时目录，不依赖固定本机状态。前端测试虽少，但类型、Lint 和生产构建均纳入 CI。

## 6. 问题与风险

### P0 / 严重

本次没有发现必然导致所有数据丢失、远程未认证代码执行或所有部署不可用的 P0 问题。

### P1 / 高风险

#### P1-1 Host Provider 绕过任务和全局 Token 预算

`TaskService.drive` 只对 `generic-command` 调用 `estimate_tokens`，其他 Provider 的 estimate 被硬编码为 0（`runtime/task_service.py:60-65`）。`BudgetManager.ensure` 对 `estimated_tokens <= 0` 直接返回（`runtime/budget.py:19-23`）。因此 Codex、Cursor、simple-worker 在派发前没有预算门。

本次用 Fake Host Bridge 实际复现：两个 `token_budget=0` 的 Codex 任务都进入 running 并创建 Host Job。现有预算测试使用绕开 ProviderPool 的测试专用 `TokenProvider`，不能覆盖生产 Host 路径。

影响：任务预算和全局日预算不能阻止真实模型费用；“超预算硬停止”仅是部分设计目标。

#### P1-2 Convention 精炼不受统一预算约束且不进入总账

`ProviderPool.refine_convention` 直接调用 Bridge，既不调用 `BudgetManager.ensure`，也不写 `token_usage`；它只写 `convention_refinements`（`providers/pool.py:123-161`）。`BudgetManager.summary` 仅汇总 `token_usage`（`runtime/budget.py:54-77`）。

影响：UI 显示的总 Token、项目/任务用量和全局日预算会漏掉明确标注“计 Token”的精炼调用。

#### P1-3 最大并发只限制“每次新增”，不是系统在途总量

Scheduler 使用 `runnable[:max_parallel_workers]`（`runtime/scheduler.py:67-84`），没有先统计 running/verifying/stopping/active Host Jobs。

本次复现：设置 `max_parallel_workers=1`，第一个 Tick 启动一个长期 Host Job，第二个 Tick 又启动另一项目任务；最终两个任务同时 running，active Host Job 为 2。

影响：长期 Host 任务可随每次 Cron Tick 持续累加，造成 CLI 进程、Token、CPU 和 API 配额失控。

#### P1-4 Permission 是“可记录”而非“可执行”的安全边界

`PermissionRepository.is_allowed` 已实现（`store/permission_repository.py:44-57`），但全后端没有生产调用；权限 API 只负责 grant/list/revoke（`api/app.py:244-257`）。任务领取、Generic Command、Host Bridge、网络要求和 secret reference 均不查询权限。

影响：deny 记录不会阻止任何动作，`audited_permissions=True` 容易被误读为强制授权。当前只能称为“权限决策台账”。

#### P1-5 完成证据可能是陈旧文件，evidence hash 不绑定 artifact 内容

Verification Engine 只检查进程返回码、文件是否存在、是否包含字符串（`runtime/verification.py:27-53`）。它不记录执行前快照、mtime、inode、Git diff 或文件内容哈希。evidence hash 只包含 returncode、failure_class 和检查结果（`runtime/verification.py:55-64`）。

因此，命令即使没有产生任何新结果，只要工作区已有匹配文件，也可被标为 completed；文件之后被修改，原 evidence hash 仍不能证明当时的具体内容。

影响：核心“只有验证证据才能完成”的形式成立，但证据的来源性、完整性和可复验性不足。

#### P1-6 Context 截断会优先丢失更具体的安全规则和完成规则

Context 按 Objective、Role、global Convention、project Convention、task Convention、Boundaries、Completion rule 的顺序拼接（`runtime/context.py:34-51`），超限时只保留前缀（`runtime/context.py:93-105`）。

一个很长的全局 Convention 可以让 project/task 级约束、项目边界和“只能由验证证据完成”全部被截掉。现有测试只验证字节上限和截断标记，没有验证必须保留尾部安全段。

影响：越具体、越接近任务的规则反而更容易消失，违反设计中的作用域优先级和安全边界。

### P2 / 中风险

#### P2-1 strict 的“独立审查”并不独立

strict 路径只是再次创建 `VerificationEngine()` 并对相同 execution/specs 运行同一实现（`runtime/task_service.py:234-249`）。两次结果在确定性条件下必然相同；同一代码缺陷会同时影响两次。balanced 的 plan 也只是写入包含 objective 和 `model_tokens: 0` 的记录（`runtime/task_service.py:74-78`），没有产生计划或执行约束。

影响：质量档位在数据库中有不同 run shape，但实际质量增量远低于命名暗示。

#### P2-2 `FaultPolicy` 没有接入生产控制流

`runtime/fault_policy.py` 有清晰分类和充分参数化测试，但 `TaskService`、Scheduler 和 Recovery 都不调用它。当前实际失败行为主要由 Verification 结果、attempt 数和同证据计数决定；provider_auth、permission_denied、budget_exceeded 等不会按 FaultPolicy 自动进入 needs_human。

影响：运行手册描述的故障决策与真实状态迁移可能不一致。

#### P2-3 “无进展”计数没有独立语义

`TaskRepository.finish` 把 `no_progress_count` 直接设置为 `same_failure_count`（`store/task_repository.py:307-315`）。它没有比较 workspace diff、artifact hash、验证检查变化或模型输出。

影响：同失败与无进展是两个配置项和两个 UI 指标，但实际是同一信号，运维调参会产生错误预期。

#### P2-4 全局预算预检存在并发超额窗口

`BudgetManager.ensure` 先用独立读连接求当天总量，再在之后的任务执行和另一个事务中记账（`runtime/budget.py:24-52`）。多个并行任务可同时看到剩余额度充足并一起派发。

影响：即使未来 Host Provider 有估算值，全局日预算仍不是严格原子配额。

#### P2-5 无 PID 的 dispatch uncertainty 可能永久卡住

Bridge 重启时，`dispatching` 且无 PID 的记录转成 `recovery_hold` + `dispatch_outcome_unknown`（`host_bridge.py:340-355`）；`_refresh` 对该状态直接原样返回（`host_bridge.py:380-387`）。重复 start 因稳定 job id 只返回旧记录，cancel 又需要可转为整数的 PID。

影响：该任务、Worker 和资源锁可能无限续期，运行手册“恢复 Bridge 后 Tick 即对账”对这一分支不成立，需要人工数据库/状态文件处置，而文档没有安全流程。

#### P2-6 前端不是实时观察面

后端提供 SSE（`api/app.py:350-368`），但前端没有 `EventSource`；tasks、usage、audit、outbox 和 health 大多只在首屏加载或手动操作后局部刷新（`web/src/App.tsx:82-96`）。

影响：无人值守运行时页面可能长期显示陈旧状态、Token 和人工介入提示。

#### P2-7 非 loopback 模式与 Web 客户端不闭环

后端在非 loopback 模式要求 Bearer token 并检查 Origin（`api/app.py:135-146`），但前端 request helper 不支持配置或附加 Authorization（`web/src/api.ts:157-168`）。

影响：若用户按后端能力启用非 loopback，随镜像打包的 Web 控制台无法正常调用 API。当前 Compose loopback 部署不受影响。

#### P2-8 `container_loopback` 是信任环境开关，不是网络事实

`Settings.is_loopback` 在 `container_loopback=True` 时无条件返回 true（`config.py:28-37`）。镜像默认设置该变量并监听 `0.0.0.0`（`Dockerfile:12-16,39-40`）。Compose 的确只映射宿主 loopback，但若镜像在其他编排中直接暴露端口，认证会被关闭。

影响：安全性依赖部署者保持 Compose 网络约束；变量命名容易让其他部署误用。

#### P2-9 设计文档和当前实现存在明显漂移

主设计仍写有“状态：重写基线草案”，同时包含已废止的宿主 scheduler 设计；它还描述可配置 Role、Step、Artifact、Delta handoff、cached token、cost、修改 Goal/预算、接受当前结果等未实现能力。README 又直接称 Sprint 0–9 已完成。

影响：验收人员容易把设计目标、历史方案和当前能力混在一起。

### P3 / 一般风险与工程债

1. **前端测试过少**：只有 5 项导航/文案测试；创建任务、错误处理、revision 冲突、控制动作、Provider 切换、设置和 Convention 均未覆盖。
2. **关键执行代码覆盖率低**：Host Bridge 56%，simple-worker 49%；真实 HTTP handler、超时、输出截断、信号失败、state corruption、API 错误和多种工具路径存在明显空白。
3. **CI 未构建 Docker 或运行真实 HTTP E2E**：`.github/workflows/quality.yml` 只跑 Python/前端门，未执行 `docker compose config`、镜像构建、`scripts/e2e_release.py` 或生产依赖审计。
4. **Python 无锁文件**：`pyproject.toml` 只有范围约束；前端 manifest 大量使用 `latest`。虽然 pnpm lock 已提交，但 Python 和基础镜像仍不能完全重复解析。
5. **基础镜像使用可变 tag**：Dockerfile 写 `node:22-alpine` 和 `python:3.13-slim`，本次构建日志解析到 digest，但仓库未固定 digest。
6. **网络可用性判定较粗**：只探测两个 DNS IP 的 TCP/53（`runtime/connectivity.py:18-41`），不能证明目标模型 API、DNS、TLS 或代理实际可用。
7. **审计并非不可变**：SQLite 中的 audit log 对拥有数据卷写权限的用户可修改；没有 hash chain、签名或外部只追加存储。UI 文案“本地不可变审计”过强。
8. **敏感信息策略仅部分覆盖**：Provider key 只存环境变量名是优点，但 objective、Convention、命令输出和 simple-worker session 仍可包含敏感文本；Redactor 只识别 Bearer、`sk-` 和私钥块。
9. **Context Pack 留存无清理策略**：按内容哈希长期保存，session 归档也持续增长；未看到容量上限、保留期或磁盘告警。
10. **任务列表固定最多 100 项**：Repository `list` 默认 limit 100，API 无分页参数；长期运行后旧任务不可通过 API/UI访问。

## 7. 安全边界评估

### 7.1 已有有效控制

- 默认服务与 Compose 发布均为 loopback。
- 非 loopback 时使用 constant-time Bearer 比较并拒绝跨 Origin。
- Host Bridge token 最低 24 字符，project root 经过 resolve 后校验。
- Host adapter 类型固定，Codex 使用 workspace-write + never approval，Cursor 启用 sandbox。
- Generic Command 使用 argv + `shell=False`，子进程环境为 allowlist。
- Host/simple-worker 状态文件使用收敛路径，写入采用临时文件 + replace。
- 容器非 root、只读 rootfs、capabilities 全丢弃。
- 备份恢复限制为 basename，并执行 SQLite integrity check。

### 7.2 必须明确的信任假设

- Generic Command **不是沙箱**。它允许 `/bin/sh -c ...`、Python `-c` 等任意可信任务载荷，`CommandPolicy` 只检查参数中显式的绝对路径和 `../`，无法限制命令语义。
- Host CLI 拥有项目 workspace-write；工具运行的项目代码也可能执行任意逻辑。项目内容必须可信。
- 本地 loopback API 没有用户身份/RBAC；同机能访问端口的进程可创建命令任务。
- Host Bridge 默认绑定 `0.0.0.0`，安全性完全依赖高熵 token 和主机防火墙。
- simple-worker 的 `npm`、`pnpm`、`make`、`pytest` 等 allowlist 命令仍可通过项目脚本执行任意项目代码；这是工作区执行能力，不是敌对代码隔离。

### 7.3 待验证安全事项

- 真实 Codex/Cursor CLI 当前版本是否仍接受这些参数，resume 时是否保持同等 sandbox/approval 边界。
- Cursor `--trust --force` 与 `--sandbox enabled` 的组合在当前版本中的准确安全语义。
- CLI JSON stream 中是否可能输出未被当前 Redactor 捕获的密钥格式。
- Docker Desktop 下 Host Bridge 的网络可达范围及本机防火墙行为。
- Python 生产依赖和基础镜像 OS 包的已知漏洞。

## 8. 可靠性、可维护性、可测试性与扩展性

### 8.1 可靠性

优点：事务领取、revision、idempotency、lease、resource lock、WAL、backup 和 Host Job 双账本构成了较完整的单机可靠性骨架。

短板：预算和并发不是硬约束；无 PID hold 缺少退出路径；证据不绑定本次产物；Provider 探测每个 Tick 都并行执行，探测延迟会占用全局 scheduler lease；任务 lease 只由 Host Job 定期续期，容器内长命令则同步阻塞一个 Tick 线程。

### 8.2 可维护性

Repository/API 命名一致，数据访问集中，测试可读性好。主要债务是：

- `api/app.py` 路由和装配过于集中；
- 设计文档没有“当前实现/未来目标”的机器可追踪清单；
- quality profile、FaultPolicy、Permission 等名称强于实际语义；
- 多处返回松散 `dict[str, Any]`，跨层契约依赖运行时键名。

### 8.3 可测试性

后端临时数据库与 Fake Bridge 设计使核心流程易测，这是明显优点。需要补足：

- Host Provider 的真实 `TaskService + ProviderPool + BudgetManager` 集成测试；
- 已运行 Host Job 计入全局并发的测试；
- Convention refinement 计账/预算测试；
- 权限 deny 的执行阻断测试；
- stale artifact 不能完成的测试；
- Context 截断必须保留安全尾段的测试；
- dispatching/no-PID 恢复和取消测试；
- 前端 SSE/轮询与错误状态测试。

### 8.4 扩展性

ProviderRepository 和 Bridge adapter 为新增执行者提供了清晰入口，但当前 Host Bridge 只接受三个 adapter，且所有配置/能力仍是字符串字典。SQLite + 单全局 Tick 适合个人/小团队；当任务量、长任务和事件量上升时，固定扫描 1000 ready tasks、SSE 每客户端每 2 秒查库、任务列表 100 条、单数据库写锁会成为瓶颈。项目当前明确不做多租户和分布式系统，因此这不是当前缺陷，但应保持产品边界。

## 9. Token 与无人值守机制专项结论

### 9.1 已实现

- 调度、Cron、网络探测、恢复和 Generic Command 本身不调用模型。
- Task execution Token 可按 run_id 幂等写账。
- Context Pack 有确定性字节上限和内容哈希。
- Host CLI external session id 可持久化并在中断后复用。
- attempt、同失败和退避能阻止一部分无限重试。
- 人工 pause/resume/cancel/needs_human 有状态门。

### 9.2 部分实现

- “0 Token 控制链”是代码约定与返回常量，不是对所有出站模型调用的统一拦截/计量。
- Context 只包含 Objective、Role 和 Convention，不是设计中的完整最小增量 Capsule。
- Token parser 对 stream 中 usage 取最大值而非求和（`host_bridge.py:529-541`）；是否正确取决于厂商事件是累计值还是增量值，尚未用真实 CLI 验证。
- simple-worker 有 24 turn 上限，但外部中断可无限次不消耗 attempt 地重排；需要独立的 interruption/recovery budget。
- task attempt 预算覆盖执行失败，但 Convention refinement、plan 记录和所谓 independent review 不共享实际模型预算。

### 9.3 未实现或不成立

- Host Provider 派发前硬 Token 预算；
- 原子全局 Token 配额；
- cached token、成本、重复上下文避免量；
- 没有新 evidence 时禁止再次调用模型的统一门；
- permission 与预算共同作为 completed 前置条件；
- 自动按阈值轮转真实 CLI session；
- 全系统在途任务的硬并发上限。

## 10. 按优先级排序的改进建议

### 立即处理（P1）

1. **建立统一 Model Run 入口和原子预算预留**：所有 Host execution、Convention refinement、未来 review 都必须先在数据库事务中 reserve estimated token/cost；完成后 reconcile actual，失败释放或记账。`token_budget=0` 必须在领取前阻止模型 Provider。
2. **把并发限制改为系统在途配额**：在同一事务中计算/维护 active execution slot，`max_parallel_workers - active_slots` 才是本 Tick 可新增数；Host Job、running、stopping、verifying 都应占槽。
3. **把 Permission 变成执行门或更名**：在 task claim/dispatch、network、secret reference、project write 前执行 deny-overrides-allow 判定；若近期不实施，应将 UI/API 能力明确标为“决策记录，不强制执行”，并移除误导 capability。
4. **强化验证证据来源性**：至少记录执行前/后 artifact metadata 和 SHA-256；file check 的 evidence hash 应包含实际文件哈希、大小、mtime、Git diff/commit（按任务类型选择），并证明变化发生在本 run。
5. **安全地编译 Context**：Objective、task/project safety、Boundaries 和 Completion rule 应作为不可截断保留段；对 Convention 分配分层预算，优先保留更具体作用域，并在丢弃任何规则时显式阻止派发或记录警报。

### 下一迭代（P2）

6. 重新定义 quality profiles：balanced 生成可执行的确定性 plan 数据；strict 使用独立实现/进程、不同证据或真正的受预算审查器，不要把相同函数重复调用称为 independent review。
7. 将 FaultPolicy 接入任务失败状态机，并为 provider auth、permission、budget、offline、timeout 建立端到端状态测试。
8. 为 `dispatch_outcome_unknown` 设计有审计的人工解决动作：确认未启动、确认已终止、隔离并放弃；动作必须带 fencing/session generation，不能要求直接改数据库。
9. 增加独立 interruption budget 和最长 recovery_hold 时间；达到阈值进入 needs_human，而不是无限免 attempt 重排/续租。
10. 前端接入 SSE 或有退避的轮询，实时刷新 task、usage、health、outbox；断线后按 sequence 补读。
11. 非 loopback 模式提供安全的前端认证方案，或明确删除/禁止该部署模式；`container_loopback` 改成更难误用的部署断言并启动时校验。
12. 将 Convention refinement Token 合并进统一 usage，UI 分开展示 execution/refinement/review/control。

### 工程治理（P3）

13. CI 增加 Docker build、Compose config、真实 HTTP E2E、关键定向回归和依赖漏洞扫描；对覆盖率设置分模块门槛，优先 Host Bridge/simple-worker。
14. 为 Python 生成受审查锁文件，固定构建工具和基础镜像 digest；前端 manifest 避免 `latest`。
15. 拆分 `api/app.py` 为 composition root、middleware 和按领域 router；用 TypedDict/dataclass/Pydantic model 替代关键跨层 `dict[str, Any]`。
16. 为任务、audit、outbox 增加分页/保留策略；为 context/session/backup 增加容量与清理策略。
17. 重写主设计文档：每项能力标注“已实现/部分实现/设计目标/废止”，Acceptance Matrix 逐条链接到测试函数和最新执行证据。
